# Architecture

This system is a minimal webhook-driven voice loop built around Twilio, FastAPI, ngrok, and OpenAI. `run_webhook.py` starts a local FastAPI server, creates a public ngrok URL, and places an outbound Twilio call to the assessment line. Twilio fetches call instructions from `/incoming-call`, where the app initializes per-call state (turn count, silence count, scenario, and recent history) and returns TwiML `<Gather>` to listen for speech. On each turn, Twilio sends recognized speech (`SpeechResult`) to `/heard`; the app logs it, generates the next patient response with OpenAI using the scenario prompt plus recent conversation history, and returns TwiML `<Say>` + `<Gather>` for the next turn.

The design choices prioritize reliability and clarity over complexity. State is stored per `CallSid` in memory to keep the implementation simple and fast to iterate, while all conversation evidence is persisted to transcript files with UTC timestamps and role labels (`SYSTEM`, `AGENT`, `PATIENT`) for auditability and bug reporting. Guardrails include max-turn limits, silence handling, and goodbye detection to prevent runaway calls. This architecture was chosen because it is the smallest practical setup that still supports true real-time back-and-forth testing, reproducible transcripts, and scalable scenario runs for QA analysis.
